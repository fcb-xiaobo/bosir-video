# Real Video API Integration Design

## Goal

Replace the current static mock video data with real XiaoBo video content so the Windows client can:

- Load real home video content.
- Search real video content.
- Open real video details.
- Load real episode/source data.
- Request a real playable URL and play it in the Windows client when Chromium supports the media format.

The first real-data pass is limited to video browsing and playback. Login, VIP entitlement changes, wallet, payment, downloads, comments, and DLNA/cast remain out of scope for this pass.

## Current State

The Windows client currently renders data from `src/data/mock.ts`. The UI shell, local navigation, detail panel, history/favorites shell, and HTML5 player are working, but the content is static.

The packaged app black-screen issue was fixed by using relative Vite assets (`base: "./"`), and the sendable zip package is generated by `npm run dist:zip`.

| User info | `user/info` | Out of scope for first video pass except diagnostics. |

The Android code indicates that base URLs are not a simple stable literal embedded in Java/Kotlin code. The app uses app init, domain sets, TXT domain data, DoH, and native helpers such as `jni_decodeUrls` / `jni_decryptApi` in `libutil.so`. A direct probe of `https://kekys.com/...` did not reach the endpoints, so the Windows app must not hardcode that as the final API host.

## Architecture

Use an Electron main-process API gateway.

Renderer responsibilities:

- Render loading, content, empty, and error states.
- Call typed preload APIs rather than directly calling remote hosts.
- Map user actions to API calls: home load, search, detail, episode selection, play.
- Keep local UI-only history/favorites unless authenticated endpoints are added later.

Preload responsibilities:

- Expose a narrow `xiaoboApi` bridge:
  - `getApiStatus()`
  - `setApiBaseUrl(baseUrl)`
  - `loadHome()`
  - `searchVideos(keyword)`
  - `loadVideoDetail(vodId)`
  - `loadEpisodes(params)`
  - `loadPlayUrl(params)`

Main process responsibilities:

- Store API settings locally under Electron `userData`.
- Own all network calls and headers.
- Try configured/candidate base URLs.
- Normalize Android response shapes into Windows renderer models.
- Log request failures to `xiaobo-windows.log`.
- Return structured errors that can be shown on the diagnostics page.

## API Configuration

The app will support three base URL sources, in this order:

1. User-configured base URL from diagnostics page.
2. Candidate list stored in app settings.
3. Future automatic domain discovery from Android init/domain logic.

For this pass, the diagnostics page must allow manual base URL input because the stable API host is not yet proven. The implementation must still include the structure needed for automatic discovery later.

Base URL validation:

- Must start with `http://` or `https://`.
- Must end with `/` internally after normalization.
- Diagnostics should test:
  - `v5/config/appInit.capi`
  - `v6/vod/home.capi`

## Request Behavior

Default headers:

- `User-Agent: okhttp/4.12.0`
- `X-CDN: 1` for `.capi` endpoints that require it.
- `Accept: application/json, text/plain, */*`

The gateway should support request timeouts and clear error messages:

- `NO_BASE_URL`
- `NETWORK_ERROR`
- `HTTP_STATUS`
- `INVALID_JSON`
- `EMPTY_DATA`
- `UNSUPPORTED_PLAY_URL`

No payment, wallet, or entitlement mutation endpoint should be called in this pass.

## Data Mapping

Windows model `VideoDetail` should be populated from Android video models where possible:

- `id`: `id` or `vodId`
- `title`: `title`
- `subtitle`: channel/year/label summary
- `cover`: `image`, `imagePath`, or static image URL
- `summary`: `summary` or generated fallback
- `tags`: labels, channel, area, language, year
- `sources`: Android `playSources` and episode data
- `playUrl`: value returned by `vod/episodePlayUrl`, with fallback to direct episode URL if present

Because Android response wrappers vary, the parser should accept:

- `{ data: ... }`
- `{ result: ... }`
- `{ items: [...] }`
- direct arrays
- nested home blocks containing vod items

Unknown fields must not crash the app.

## UI Changes

Home/Search:

- Replace direct `mockVideos` reads with API-backed state.
- Show loading state while requesting.
- Show API error with a short retry action.
- Show "using fallback demo data" only when the real API fails and fallback is enabled.

Detail:

- Selecting a card should call `loadVideoDetail(vodId)`.
- If detail fails, show the list item data and error state.

Episodes/Playback:

- If detail includes sources/episodes, show them immediately.
- If source selection needs more data, call `loadEpisodes`.
- When an episode is selected, call `loadPlayUrl`.
- If Chromium cannot play the returned format, show a clear unsupported-format message.

Diagnostics:

- Show current base URL.
- Allow editing and saving base URL.
- Add buttons:
  - Test app init
  - Test home
  - Test search
- Show last request URL, status, duration, and error.
- Show log file path.

## Fallback Policy

Mock data remains only as a development fallback.

Production UI must make the data source visible:

- `真实接口` when data came from API.
- `演示数据` when fallback is being shown because the API failed or no base URL is configured.

Fallback must never hide API failure details in diagnostics.

## Verification

Minimum verification for this pass:

- `npm run build` succeeds.
- `npm run dist:zip` succeeds.
- Packaged zip is extracted and launched.
- Diagnostics page can save a base URL and run endpoint tests.
- If a valid base URL is configured, home loads non-mock video items.
- Search returns non-mock results for a keyword.
- Detail loads for a selected real video.
- Episode/play URL request is attempted and either plays or displays a precise unsupported/error message.

## Risks

- Base URL discovery may require deeper native/domain logic or a runtime capture from Android.
- Endpoints may require additional device headers or encrypted parameters.
- Playback URL may be HLS, redirected, tokenized, or unsupported by Chromium.
- Some content may require login/VIP; this pass should report the API error instead of bypassing entitlement rules.

## Out Of Scope

- Bypassing login, VIP, payment, or entitlement checks.
- Wallet or recharge flow.
- Download/offline cache.
- DLNA/cast.
- Comment posting.
- Full Android native player parity.
